# Comparison and Ranking Pages

## Overview

Three new page types sit alongside the per-repo pages:

| Page type | Path | Generated by | Requires LLM? |
|-----------|------|--------------|---------------|
| Comparison | `docs/compare/<group-slug>.md` | `compare.yml` workflow | Yes — recommendation section |
| Group ranking | `docs/rankings/<group-slug>.md` | VitePress data loader | No — computed from frontmatter |
| Global ranking | `docs/rankings/index.md` | VitePress data loader | No — computed from frontmatter |

Ranking pages are computed at build time from existing page frontmatter — no workflow, no LLM, no extra storage. Comparison pages require LLM reasoning for the recommendation and are written as committed markdown files.

---

## `docs/schema/groups.yaml`

Source of truth for comparison group membership. Maintained manually; `compare.yml` triggers on every push that touches this file.

```yaml
version: "1.0.0"

groups:
  - id: http-clients
    label: "HTTP Clients"
    description: "Libraries for making HTTP requests in JavaScript/TypeScript"
    repos:
      - axios/axios
      - sindresorhus/got
      - node-fetch/node-fetch
```

**Rules**:
- `id` must be a URL-safe slug (kebab-case); used as the filename for `docs/compare/<id>.md` and `docs/rankings/<id>.md`
- A group must have ≥ 2 repos before `compare.yml` will generate a page for it
- Repos listed here must have an evaluated page in `docs/repos/` — `compare.yml` fails the group (not the whole run) if any listed repo is missing its page
- `version` is bumped whenever any group is added, removed, or its `repos` list changes; this triggers comparison page regeneration via `compare.yml`

**Adding a repo to a group**: edit `groups.yaml` → push to `main` → `compare.yml` re-generates the affected comparison page.

**Group discovery**: when `evaluate.ts` completes a new evaluation, it scans `groups.yaml` for groups whose existing repos share 2+ tags with the newly evaluated repo and posts a comment on the issue: *"This repo may overlap with group '[label]' (shared tags: X, Y) — consider adding to `docs/schema/groups.yaml`."* No automatic update; curation stays manual.

## Comparison Page

### Frontmatter Schema

```yaml
---
title: "HTTP Clients — Comparison"
group_id: http-clients
group_label: "HTTP Clients"
repos: [axios/axios, sindresorhus/got, node-fetch/node-fetch]
recommended: [axios/axios]           # one or more repo slugs
generated_at: 2026-03-25
schema_version: "1.0.0"             # classification.yaml version at generation time
model_id: "gpt-4o-mini"
---
```

Validated by `docs/schema/compare-page.schema.json` (generated from Zod, same pipeline as `repo-page.schema.json`).

### Body Structure

Validated by `validate-compare.ts` against `docs/schema/compare-template.yaml`.

```markdown
## Summary

| Repo | Maturity | Maintenance | Completeness | Documentation | Community | Total | Verdict | Recommended |
|------|---------|-------------|--------------|---------------|-----------|-------|---------|-------------|
| [axios/axios](../repos/axios-axios.md) | 4 | 4 | 5 | 5 | 5 | 23/25 | gem | ✓ |
| [sindresorhus/got](../repos/sindresorhus-got.md) | 4 | 3 | 5 | 4 | 3 | 19/25 | solid | |

## Recommendation

**Primary choice**: [axios/axios](../repos/axios-axios.md) — ...reason...

Use [got](../repos/sindresorhus-got.md) when: ...

## Comparison

Per-dimension narrative comparing the repos across each scoring axis.
```

The Summary table is generated by `compare.ts` from the repos' frontmatter — scores are not re-generated by the LLM. The Recommendation and Comparison sections are LLM-authored.

`Total` = sum of all dimension integer scores. `Recommended` = ✓ for repos listed in frontmatter `recommended`.

## Group Ranking Page

Generated by a VitePress data loader from `groups.yaml` + repo page frontmatter. No committed markdown — the page is built at `docs:build` time.

### Body Structure

```markdown
## HTTP Clients — Rankings

| Rank | Repo | Total | Maturity | Maintenance | Completeness | Documentation | Community | Verdict |
|------|------|-------|---------|-------------|--------------|---------------|-----------|---------|
| 1 | [axios/axios](../repos/axios-axios.md) | 23/25 | 4 | 4 | 5 | 5 | 5 | gem |
| 2 | [sindresorhus/got](../repos/sindresorhus-got.md) | 19/25 | 4 | 3 | 5 | 4 | 3 | solid |

*Last updated: 2026-03-25*
```

Sorted by `Total` descending; ties broken by `verdict` order (gem > solid > experimental > reference-only > abandoned).

## Global Ranking Page

Generated by a VitePress data loader from all `docs/repos/*.md` frontmatter. No committed markdown.

Lives at `docs/rankings/index.md`.

### Body Structure

```markdown
## All Projects — Rankings

| Rank | Repo | Category | Total | Maturity | Maintenance | Completeness | Documentation | Community | Verdict |
|------|------|----------|-------|---------|-------------|--------------|---------------|-----------|---------|
| 1 | [axios/axios](../repos/axios-axios.md) | library | 23/25 | 4 | 4 | 5 | 5 | 5 | gem |
| 2 | ... | | | | | | | | |

*N repos evaluated · last updated: build date*
```

Same sort order as group ranking. Category column links to a future tag/category filter (Phase 3).

## `docs/schema/compare-template.yaml`

```yaml
sections:
  - heading: Summary
    description: "Summary table with one row per repo"

  - heading: Recommendation
    min_content_length: 50
    description: "Primary recommendation and per-scenario guidance"

  - heading: Comparison
    min_content_length: 100
    description: "Per-dimension narrative comparing the repos"
```

## Open Questions

- Should group ranking pages be committed markdown (generated by `compare.yml`) or VitePress data loaders? **Resolved**: data loaders — all data already exists in frontmatter, no LLM needed, always in sync with the latest scores.
- Should `recommended` allow multiple repos? **Yes** — some groups have legitimately different best choices per use case.

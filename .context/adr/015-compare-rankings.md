# ADR-015: Comparison and Ranking Page Design

**Status**: Accepted
**Date**: 2026-03-25

## Context

As the evaluated corpus grows, repos that serve similar purposes (e.g. HTTP clients, test runners, bundlers) will inevitably appear. A flat list of individual repo pages is insufficient for answering "which of these should I use?" without manually cross-referencing pages.

Three distinct needs were identified:
1. **Side-by-side comparison** of overlapping repos with an explicit recommendation
2. **Ranked view** of repos within a comparison group
3. **Global ranked view** of all evaluated repos

## Decision

### Comparison groups

`docs/schema/groups.yaml` is the curated source of truth for comparison group membership. Groups are maintained manually by the operator. `evaluate.ts` surfaces suggestions (shared-tag heuristic) but does not auto-update groups.

### Comparison pages — LLM-generated, committed markdown

`docs/compare/<group-slug>.md` is generated by `compare.yml` (a new workflow). The LLM writes the Recommendation and Comparison sections using the pre-existing evaluated pages as input — score integers are copied from frontmatter directly, not re-generated. Comparison pages are committed markdown so they are auditable, diffable, and browsable without a build step.

Alternatives rejected:
- **Fully manual**: too high maintenance burden as the corpus grows
- **VitePress data loader only**: cannot produce the LLM recommendation section without a server
- **Regenerate scores per group**: wasteful; scores already exist on individual repo pages

### Ranking pages — VitePress data loaders, not committed markdown

Group ranking pages (`docs/rankings/<group-slug>.md`) and the global ranking page (`docs/rankings/index.md`) are generated at build time by VitePress data loaders from existing page frontmatter. No LLM, no workflow, no committed file — always in sync with the latest scores after any build.

Alternatives rejected:
- **Committed markdown generated by workflow**: would require a separate commit after every repo update; data loaders are simpler and always current

### Total score

`Total` = sum of all dimension integer scores. No weighting (see ADR-012 amendment: dimension weights rejected). Ties broken by `verdict` order: gem > solid > experimental > reference-only > abandoned.

## Consequences

- Comparison pages are one LLM generation away from stale when a member repo is re-evaluated; `compare.yml` re-fires on any `docs/repos/` push to `main` to keep them current
- Ranking pages are always current — they rebuild with every `deploy.yml` run
- Groups are intentionally curated, not automatically detected; this trades completeness for precision
- `compare-page.schema.json` (new generated artefact) must be regenerated when `classification.yaml` dimensions change, alongside `repo-page.schema.json`
- Adding a new dimension bumps `schema_version` and re-evaluates all repos; `compare.yml` fires after all re-evaluations complete (triggered by `docs/repos/` push), regenerating comparison pages with updated scores automatically
